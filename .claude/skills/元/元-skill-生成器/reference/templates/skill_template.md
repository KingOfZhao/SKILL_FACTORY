## Input Specification（输入规范 - 必须横竖穷举）

### 横向穷举（同一层级的多种可能形式）
- 主要输入形式：
  - 自然语言描述（e.g. “帮我生成一个登录页面”）
  - 代码片段 / 文件路径（e.g. 粘贴的 Dart 文件、项目相对路径）
  - 错误日志 / 报错信息（完整栈追踪或截图描述）
  - 配置参数（JSON / YAML / 键值对）
  - 多模态（设计稿截图、Figma 链接描述、视频帧）
  - 组合输入（e.g. “基于这个旧代码 + 这个新需求”）

- 输入来源：
  - 当前聊天消息
  - 剪贴板 / 选中代码
  - 项目文件（需 MCP）
  - 外部链接 / API 返回（需 MCP）

### 纵向穷举（不同抽象/异常层级）
- 最原始输入：纯文本、模糊一句话
- 结构化输入：JSON schema、带字段的 YAML、带占位符的模板
- 最小有效输入：最少需要哪些字段/信息才可执行
- 边缘 / 非法输入：
  - 空输入 / 仅标点
  - 格式错误（e.g. JSON 缺引号）
  - 超出范围（e.g. 文件大小 > 10MB、语言非 Dart）
  - 冲突输入（e.g. 同时要求 Material 和 Cupertino 风格）

## Output Specification（输出规范 - 必须横竖穷举）

### 横向穷举（可能的输出物类型）
- 单文件内容（纯代码 / Markdown / JSON）
- 完整项目结构（目录树 + 多文件内容）
- diff 格式（--- old +++ new）
- 可运行构建物（e.g. APK、zip 包链接建议）
- 报告 / 分析结果（表格、列表、总结）
- 后续建议（e.g. “还需要创建 tester Skill”）
- 错误 / 失败输出（带原因、修复建议）

### 纵向穷举（不同质量/状态）
- 理想输出：完整、可直接复制运行、符合最佳实践
- 最小可用输出：核心逻辑实现，但缺少优化/测试
- 带元信息的输出：包含变更说明、版本建议、局限声明
- 失败 / 部分输出：
  - 无法执行时：明确拒绝 + 原因 + 建议拆分 Skill
  - 部分成功：已完成部分 + 剩余待办列表

## 示例 Input → Output 对（至少 2-3 个，强制要求）
**Example 1**  
Input: “修复这个 Flutter 页面布局溢出问题” + 粘贴的代码  
Output: diff 格式修复代码 + 解释变更 + 建议添加的测试用例

**Example 2**  
Input: 空 或 “随便写个页面”  
Output: 拒绝 + 要求至少提供：页面类型、关键组件、状态管理偏好

## Enforcement Rules
- 如果输入不符合以上任一穷举项 → 立即回复：“输入不符合规范，请补充 [具体缺失项] 或使用配套 Skill：[建议列表]”
- 输出必须始终包含：局限声明 + 是否建议拆分 Skill
